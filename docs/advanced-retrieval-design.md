# 목차 Top1 탐색 정밀도 고도화 방안

이 문서는 현재 진행한 "LLM + 프롬프트 엔지니어링 기반 체크포인트 목차 Top1 탐색" 설계를 보강하기 위해, 더 정밀한 탐색 정확도를 만들 수 있는 추가 AI 기술을 정리한 문서입니다.

현재 방식은 빠르게 실험하고 운영 반영하기 좋다는 장점이 있습니다. 다만 Top1 정확도를 더 끌어올리려면, 프롬프트만 조정하는 방식보다 "후보 검색", "재정렬", "학습 기반 분류", "운영 피드백 반영"을 함께 설계하는 쪽이 더 효과적입니다.

## 1. 현재 방식의 한계

현재 방식:

- 도메인 정보, 사용자 질문, LLM 답변, 목차 정보, 상담 데이터 일부를 프롬프트에 넣고
- LLM이 가장 적합한 목차 또는 체크포인트를 Top1로 선택하도록 유도하는 구조

주요 한계:

- 목차 수가 많아질수록 프롬프트 안에서 모든 후보를 안정적으로 비교하기 어려움
- 비슷한 표현의 목차가 많은 경우, LLM이 표면 문구에 끌릴 수 있음
- 같은 의미를 다른 표현으로 말한 질문에 대해 일관성이 떨어질 수 있음
- 프롬프트 변경 효과가 데이터셋 전체에서 항상 재현되지 않을 수 있음
- Top1은 맞추기 어려운데 Top3 안에는 들어오는 경우가 많아, 후보 생성과 최종 선택을 분리할 필요가 있음

## 2. 권장 방향

가장 현실적인 고도화 방향은 아래 4단계 구조입니다.

1. 1차 후보 검색
2. 2차 후보 재정렬
3. 필요 시 학습 기반 분류기 추가
4. 오답 로그 기반 지속 개선

즉, "LLM이 처음부터 전체 목차를 한 번에 맞히는 구조"보다 "좁힌 후보군 안에서 LLM 또는 모델이 Top1을 고르는 구조"가 일반적으로 더 정밀합니다.

## 3. 적용 가능한 AI 기술

### 3.1 임베딩 기반 벡터 검색

개념:

- 사용자 질문, LLM 답변, 예상 질문, 목차 설명, 체크포인트 대표 문구를 임베딩 벡터로 변환
- 질의와 의미적으로 가까운 목차/체크포인트를 코사인 유사도 등으로 검색

왜 유효한가:

- 표현이 달라도 의미가 비슷하면 가까운 후보를 찾을 수 있음
- 전체 목차 중 상위 `K`개만 빠르게 추릴 수 있어, 이후 LLM 판단 범위를 줄일 수 있음

권장 적용 방식:

- 인덱스 단위:
  - 체크포인트 단위 임베딩
  - 목차 단위 임베딩
  - 질문 예시 단위 임베딩
- 질의 텍스트 구성:
  - `사용자 질문`
  - `LLM 답변`
  - 필요 시 `도메인 텍스트`
- 반환 결과:
  - Top10 또는 Top20 후보

장점:

- 구축 난도가 상대적으로 낮음
- 프롬프트 길이 문제를 완화
- Top1뿐 아니라 Recall@K 개선에 효과적

주의점:

- 임베딩만으로는 의미가 유사하지만 업무 정책상 다른 목차를 잘못 끌어올릴 수 있음
- 따라서 단독 사용보다 재정렬 단계와 결합하는 것이 좋음

## 3.2 하이브리드 검색: BM25 + Dense Retrieval

개념:

- 키워드 기반 검색(BM25)과 의미 기반 검색(임베딩 검색)을 함께 사용
- 두 점수를 합치거나, 각각의 상위 후보를 합집합으로 묶어 후보군 생성

왜 유효한가:

- 특정 상품명, 제도명, 업무 용어, 약어는 키워드 검색이 강함
- 의미 유사성은 임베딩 검색이 강함
- 두 방식을 합치면 한쪽의 약점을 다른 쪽이 보완함

추천 시나리오:

- 체크포인트 명칭이 정형적이고 용어 의존성이 높을 때
- 같은 의미라도 표현 변형이 많은 상담 질문을 함께 다뤄야 할 때

권장 구조:

1. BM25 TopN 추출
2. Dense Retrieval TopN 추출
3. 후보 합집합 생성
4. 재정렬 모델 또는 LLM으로 Top1 결정

## 3.3 Cross-Encoder 재정렬 모델

개념:

- 질의와 각 후보를 쌍(pair)으로 넣고, 해당 후보가 정답일 확률을 직접 점수화하는 모델
- 임베딩 검색보다 느리지만 보통 더 정밀함

왜 중요한가:

- 실제 Top1 정확도는 후보 검색보다 재정렬에서 크게 개선되는 경우가 많음
- "이 질문과 이 목차가 정말 맞는가"를 직접 판정하는 구조라 오분류를 줄이기 좋음

권장 적용 방식:

1. 1차 검색으로 후보 10~30개 확보
2. 각 후보에 대해 Cross-Encoder 점수 계산
3. 점수 상위 3개를 선택
4. 최종적으로 LLM 또는 규칙 기반으로 1개 결정

입력 예시:

- 질의: `사용자 질문 + LLM 답변`
- 후보: `목차명 + 체크포인트명 + 대표 질문 + 설명`

장점:

- 의미 차이를 세밀하게 판별 가능
- 비슷한 후보가 많은 환경에서 강함

단점:

- 모든 후보 쌍을 계산해야 해서 속도가 느릴 수 있음
- 운영 시 캐시, 배치 처리, 후보 수 제한이 필요함

## 3.4 Pairwise / Listwise LLM Reranking

개념:

- LLM에게 전체 후보를 한 번에 주는 대신, 상위 후보만 비교하게 하는 재정렬 방식
- 두 후보씩 비교하는 pairwise 방식 또는 소수 후보 전체를 정렬하는 listwise 방식을 사용

왜 현재 구조와 잘 맞는가:

- 이미 LLM 프롬프트 엔지니어링 자산이 있음
- 전체 목차를 한 번에 판단시키는 것보다 상위 후보 5~10개만 비교시키는 것이 안정적임

추천 방식:

- 1차 검색으로 Top10 후보 확보
- LLM에 아래 정보만 입력
  - 사용자 질문
  - LLM 답변
  - 후보 목차/체크포인트 정보
  - 선택 기준
- 출력:
  - Top1
  - 근거
  - 차순위 후보

보강 팁:

- "정답이 없으면 없음" 옵션 제공
- 반드시 선택 근거를 구조화된 필드로 받기
- 유사 후보 간 차이를 비교하도록 프롬프트 설계

## 3.5 지도학습 기반 분류기

개념:

- 과거 상담 데이터와 정답 목차/체크포인트를 이용해 supervised classification 모델을 학습

가능한 모델:

- LightGBM/XGBoost + 텍스트 임베딩 특징
- BERT 계열 분류기
- Instruction-tuned classifier

언제 고려할 만한가:

- 정답 라벨이 충분히 쌓였을 때
- 운영 트래픽이 많아 일관성과 속도가 중요할 때
- 특정 목차에 대한 반복 질의 패턴이 많은 경우

장점:

- 반복적으로 비슷한 질문이 들어오는 환경에서 매우 안정적
- 추론 비용이 LLM보다 낮을 수 있음

단점:

- 라벨 데이터 품질에 크게 의존
- 신설 목차나 정책 변경에 취약할 수 있음

현실적인 적용:

- 분류기를 Top1 단독 결정기로 쓰기보다
- "후보 생성" 또는 "재정렬 보조 점수"로 함께 사용하는 편이 안전함

## 3.6 Learning to Rank

개념:

- 질의와 후보의 관계를 바탕으로 "정답을 더 위로 올리는" 랭킹 모델을 학습

왜 적합한가:

- 현재 목표가 분류 정확도 전체보다 "Top1 정확도 향상"에 더 가깝기 때문
- 랭킹 문제는 분류보다 실제 운영 목표와 더 직접적으로 맞을 수 있음

입력 특징 예시:

- BM25 점수
- 임베딩 유사도
- Cross-Encoder 점수
- 도메인 일치 여부
- 제목 alias 일치 여부
- 예상 질문 유사도
- 과거 클릭/선택 로그

추천 방식:

- LambdaMART 계열
- pairwise ranking loss 기반 모델

## 3.7 지식 그래프 또는 규칙 기반 보강

개념:

- 완전히 AI 모델만으로 처리하지 않고, 도메인-목차-체크포인트 간 연결 정보를 규칙 또는 그래프로 관리

적용 예:

- 특정 도메인에서는 허용 가능한 목차 후보를 제한
- 특정 상품/제도 키워드가 나오면 일부 체크포인트 가중치 증가
- alias 사전, 금칙어, 배제 규칙을 별도 관리

효과:

- 모델이 흔들리는 경계 사례에서 오답을 줄임
- 운영자가 통제 가능한 안전장치 역할

추천 원칙:

- 규칙은 정답을 직접 강제하기보다 후보 축소와 점수 보정에 활용

## 3.8 불확실성 추정과 Human-in-the-loop

개념:

- 모델이 자신 없는 경우 억지로 Top1을 선택하지 않도록 설계

적용 방식:

- Top1 점수와 Top2 점수 차이가 작으면 수동 검토 큐로 보냄
- LLM 근거가 약하거나 금칙 조건에 걸리면 재시도
- 복수 모델 간 불일치 시 검토 대상으로 전환

왜 필요한가:

- Top1 정확도만 보는 시스템은 틀릴 때 비용이 클 수 있음
- "자동 처리 가능한 건 정확하게 자동 처리하고, 애매한 건 보류" 전략이 실무에서 더 유리함

## 4. 가장 추천하는 목표 아키텍처

현 시점에서 가장 현실적인 구조는 아래와 같습니다.

### 옵션 A. 하이브리드 검색 + LLM 재정렬

구성:

1. BM25 검색
2. 임베딩 검색
3. 후보 합치기
4. 상위 10개 후보를 LLM이 재정렬
5. Top1 반환

장점:

- 현재 프롬프트 자산을 그대로 활용 가능
- 개발 복잡도가 과도하게 높지 않음
- 빠르게 PoC와 운영 전환 가능

적합한 경우:

- 아직 대규모 라벨 데이터가 부족한 경우
- 우선 성능 향상을 빨리 보고 싶은 경우

### 옵션 B. 하이브리드 검색 + Cross-Encoder + LLM 최종판정

구성:

1. BM25 + Dense Retrieval로 Top20 후보 생성
2. Cross-Encoder로 Top5 재정렬
3. LLM이 Top3 중 최종 Top1 선택

장점:

- 정밀도 중심 구조
- 유사 후보가 많은 케이스에서 강함

적합한 경우:

- 탐색 정확도가 운영 KPI에 직접 연결되는 경우
- 약간의 추론 비용 증가를 감수할 수 있는 경우

### 옵션 C. 하이브리드 검색 + 학습 기반 랭커

구성:

1. BM25 + Dense Retrieval 후보 생성
2. 랭킹 모델로 Top1 산출
3. 낮은 확신 구간만 LLM 또는 사람 검토

장점:

- 운영비와 응답속도에 유리
- 데이터가 누적될수록 성능이 안정화될 가능성이 큼

적합한 경우:

- 장기적으로 대량 운영을 고려하는 경우

## 5. 샘플 검증 설계 제안

현재의 샘플 검증을 더 정밀하게 만들려면, 단순 정답/오답 확인보다 아래 지표를 함께 보는 것이 좋습니다.

### 5.1 평가 지표

- Top1 Accuracy
- Top3 Recall
- MRR
- nDCG
- 도메인별 Accuracy
- 목차별 Accuracy
- `UNKNOWN` 또는 애매 구간 분리 정확도

왜 필요한가:

- Top1 Accuracy만 보면 후보군은 잘 뽑았는데 최종 선택만 틀리는지 구분하기 어려움
- Recall@K를 같이 보면 "검색 문제"인지 "재정렬 문제"인지 분해 가능함

### 5.2 샘플셋 구성

샘플셋은 아래 유형으로 나누는 것이 좋습니다.

- 명확한 정답이 있는 쉬운 케이스
- 비슷한 목차가 여러 개 있는 혼동 케이스
- 질문은 짧지만 답변 문맥이 중요한 케이스
- 사용자 질문만 보면 애매하고 LLM 답변을 함께 봐야 하는 케이스
- `UNKNOWN` 처리되어야 하는 케이스
- 신규 표현, 오탈자, 구어체가 섞인 케이스

### 5.3 A/B 실험 단위

비교할 실험군 예시:

- Baseline: 현재 프롬프트 단독 방식
- 실험 1: Dense Retrieval + 프롬프트
- 실험 2: Hybrid Retrieval + 프롬프트
- 실험 3: Hybrid Retrieval + Cross-Encoder
- 실험 4: Hybrid Retrieval + Cross-Encoder + LLM rerank

기록 권장 항목:

- 질의 ID
- 정답 목차/체크포인트
- Top1 예측
- Top3 후보
- 각 단계 점수
- 오답 유형
- 수동 분석 메모

## 6. 오답 분석 체계

정확도를 올리려면 모델 교체보다 오답 유형 분류가 더 중요할 수 있습니다.

권장 오답 유형:

- 후보 검색 실패
- 후보는 맞았지만 재정렬 실패
- 도메인 자체가 잘못 들어감
- alias/동의어 미반영
- 질문보다 답변 문맥이 더 중요했는데 반영 부족
- `UNKNOWN`이어야 하는데 기존 목차로 강제 매핑
- 라벨 자체 품질 문제

이 분류가 있어야 다음 액션이 선명해집니다.

- 검색 개선이 필요한지
- 프롬프트 개선이 필요한지
- 학습 데이터 보강이 필요한지
- 규칙 추가가 필요한지

우선순위는 아래 순서를 권장합니다.

1. 현재 방식 유지
2. 임베딩 검색 추가
3. BM25와 결합한 하이브리드 검색 도입
4. Top10 후보 기반 LLM 재정렬로 전환
5. 오답 로그 누적 후 Cross-Encoder 또는 랭킹 모델 도입

즉, 가장 먼저 시도할 만한 개선은 아래입니다.

- 전체 목차를 한 번에 프롬프트에 넣지 않기
- 벡터 검색 또는 하이브리드 검색으로 후보를 먼저 줄이기
- LLM은 최종 선택기 역할로 좁혀 쓰기
- 평가 지표를 Top1 Accuracy만 보지 말고 Recall@K까지 함께 보기

## 8. 권장 결론

정밀한 Top1 목차 탐색이 목표라면, 프롬프트 엔지니어링만으로 밀어붙이는 구조보다 아래 조합이 더 적합합니다.

- 단기: `Hybrid Retrieval + LLM Reranking`
- 중기: `Hybrid Retrieval + Cross-Encoder Reranking`
- 장기: `Hybrid Retrieval + Learning to Rank + Human-in-the-loop`

현재 자산을 최대한 활용하면서도 성능 개선 가능성이 큰 시작점은 다음 한 줄로 요약할 수 있습니다.

"LLM 단독 분류"에서 "검색 기반 후보 축소 후 LLM 재정렬" 구조로 바꾸는 것이 가장 효과적입니다.
